                           upload Q4_K to device, set output_dtype_ = kQ4_K

Algorithm: greedy Q4_K quantization (no iterative refinement of d/m pairs).
  - dequant Q6_K block to fp32
  - per 32-elem sub-block: d_i = (max - min)/15, m_i = -min (forced m≥0 via
    "if (mn > 0) mn = 0")
  - super_d = max(d_i), super_dmin = max(m_i), encode each as 6-bit
  - pack scales[12] per Q4_K bit-packing layout
  - re-quantize using rounded-back scales so reconstruction matches kernel

Likely bug surfaces (un-debugged at revert time):
  1. host_fp32_to_fp16: my round-to-zero version may produce unintended
     Inf for fp32 values that round to fp16 max-exponent in edge cases.
  2. lm_head Q6_K likely has blocks with values outside the dynamic range
     a single fp16 super-scale can represent in Q4_K format (Q4_K's range
     = 15 × max_d, while Q6_K's range = 32 × max_int8 × max_d ≈ 4096×).
     Some lm_head blocks may genuinely require iterative refinement to
     fit cleanly in Q4_K's 4-bit quantization grid.
  3. The "if (mn > 0) mn = 0" greedy simplification can over-extend the
     quantization range for sub-blocks with all-positive narrow data,
     wasting precision and potentially producing weights that overflow
     when multiplied by activations into fp32 logits.
  4. Greedy d_i/m_i selection (no make_qkx2_quants iterative search)
     accumulates error worse than llama.cpp's algorithm — could push
     individual quants over the saturation boundary.
