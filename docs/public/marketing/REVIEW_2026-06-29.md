## BLOCKING / should-fix before launch

### B1 — README crown row is a refutable decode overclaim (NOT a gpt-oss claim, NOT in ledger)
`README.md:11` states the daily-driver crown (`qwen35moe`) is **"pp512 + tg128 faster than llama.cpp SYCL master."**
The freshly-added SUPERSEDED banner in the same repo (`docs/PUBLISHING_READINESS_2026-06-21.md:6`) states
**"Crown decode is now parity, not a win."** That is a direct, skeptic-refutable contradiction sitting in the
launch-facing README. The crown competitive throughput is not in the ledger (the ledger only covers crown PPL gate 6.4527).
**Fix:** soften the README crown row to "prefill faster vs llama.cpp SYCL master; decode parity" (per the banner), or
hold a fresh same-box crown bench and add it to the ledger before claiming a crown decode win.

### B2 — README carries other un-ledgered, un-re-verified competitive perf claims
`README.md:12,16,19` quote `qwen3`/`qwen2` "prefill +15% vs llama.cpp SYCL," 27B "prefill faster than llama Vulkan,"
72B "1.44× faster decode" (that one is vs our own layer-split, internal, OK), crown "faster than SYCL master."
DOC_HYGIENE_CHANGES explicitly left these "as-is" (out of sweep scope). None are in the launch ledger; a B70 owner
could test the +15% qwen3 line directly. **Fix:** either reconcile against `COMPETITIVE_SCORECARD_2026-06-25.md` or
add a one-line "pre-ledger, pending re-bench" caveat to the non-gpt-oss rows before the README is treated as launch copy.

### B3 — Standing pre-publish gate (already disclosed; must remain blocking)
Every artifact correctly flags it, but for the record the two hard gates are still UNMET in-session:
(1) clean-box re-bench of the 20b head-to-head vs *current* llama-SYCL (the 927/927/896 + 50.3/49.9/49.4 baseline is

presented as "fresh as of launch day" until these run.
