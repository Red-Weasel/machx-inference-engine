### Body ("Show HN", engineering-first)

This is a from-scratch C++/SYCL inference engine for Intel Arc (Battlemage / BMG-G31) — no CUDA, no ported llama.cpp kernels in the hot paths. The goal has been narrow and specific: bring up brand-new model architectures *fast* on Arc, and be honest about where we win and lose. gpt-oss is the 8th architecture; it has three pieces most engines don't special-case (per-head attention **sinks**, alternating **sliding-window/dense** layers, and **MXFP4** experts) on top of a top-4 MoE.

Two things are worth your time even if you don't own an Arc card: a small-M XMX MoE bug hunt, and a tensor-parallel run that turned out to be comm-bound, not compute-bound.

**gpt-oss-20b, single B70, vs llama.cpp SYCL (same GGUF, llama's *best* fa-on config).** We beat it on both axes and the gap is flat across context:

| context | engine prefill | llama prefill | speedup | engine decode | llama decode | speedup |
|--------:|---------------:|--------------:|:-------:|--------------:|-------------:|:-------:|
| 512  | **1795** t/s | 927 t/s | **1.94×** | **58.3** t/s | 50.3 t/s | **1.16×** |
| 2048 | **4147** t/s | 927 t/s | **4.47×** | **57.4** t/s | 49.9 t/s | **1.15×** |
| 4096 | **3428** t/s | 896 t/s | **3.83×** | **55.6** t/s | 49.4 t/s | **1.13×** |

Reproduce: ours `ie-bench --gguf <model> --prefill T --decode N --warmup 2` (median); llama `ONEAPI_DEVICE_SELECTOR=level_zero:0 llama-bench -m <model> -ngl 99 -fa 1 -p T -n 0` and `... -n 128 -d {depth}`. Same 11.27 GiB `gpt-oss-20b-mxfp4.gguf` (20.9 B / 3.6 B active), single 32 GB card both sides. The decode margin (1.13–1.16×) is thin and within this box's noise — we call that out rather than rounding it up.

**The prefill win was a self-inflicted wound we found by profiling.** The attention q/k/v/o projections *and* the 32-expert MoE router were silently falling out of the batched-GEMM path and running as **per-token GEMVs** — **196,608 GEMV launches in a single 2048-token forward, ~88% of prefill** — because our batched path was gated on dimensions divisible by 256/64, which gpt-oss's hidden=2880 and its 32-way router both fail. Routing them through batched fp16 GEMMs (oneDNN handles those shapes fine) flipped prefill from a loss to a win: **759 → 4147 t/s @2K, a 5.5× self-improvement.**

**gpt-oss-120b on two Arc Pro B70s (tensor-parallel).** There is no working P2P on these cards, so every all-reduce bounces through host RAM — and that, not quant or VRAM spill, dominates. Profiling showed **~68% of prefill was two host-bounced all-reduces**; the attention all-reduce is pure head-shard overhead, so we *replicate* attention across both cards instead of sharding it, dropping one all-reduce: **433 → 679 t/s** prefill at moderate context. Decode lands at **~31 tok/s** (peak measured 32.05, up 2.24× from 14.33 after an async-scatter all-reduce). The model loads display-safe (card0 29.3 GB VRAM + 3.4 GB host spill).

**Two bugs worth the writeup:**

- **fp16 residual overflow.** 120b chat produced `"!"` then NaN garbage. Activations exceed fp16's max (65504) around layer 35; one `inf` poisons the KV cache, NaNs cascade, argmax collapses to token 0 (`"!"`). Saturating the residual add to ±65504 fixes it and is an exact no-op for in-range models (crown PPL stays bit-exact at 6.4527).
- **small-M XMX MoE corruption.** Batched prefill produced garbage (PPL ~123). With E=128 experts and top-4, each expert sees only ~1–8 rows per step, and BMG's f16 XMX path corrupts every local row *except row 0* at small M — so token 0 stayed bit-exact while everything after it garbled, which is what gave it away. Padding small experts up to M=32 fixed it: batched PPL **123 → 15.1985, bit-identical to the T=1 path (15.1985)**.

**Correctness gating.** Every engine change holds a crown-model PPL gate bit-exact (6.4527, NLL 1.864495); the 120b correctness claim is the batched-vs-T=1 PPL being bit-identical (15.1985). The int-dot decode paths (W4A8/W8A8) move PPL < 0.1%. OpenAI Harmony tool-calling is wired end-to-end (render + parse + multi-turn, zero server changes) and crown-neutral.

**Honest losses / caveats.** This is not a general "best engine for Intel GPUs" claim — it's "fastest *known* gpt-oss on Intel Arc, beats llama.cpp both axes on the 20b, same-hour measured." On dense-attention decode we don't win — e.g. Qwen3-Coder-30B decode is ~0.65× current llama (50.5 vs 77.3 t/s); dense decode is our known weak spot (the scoped fix is MTP self-spec decode). 120b decode carries ±15% box noise (same binary has measured 12–32 t/s), and the long-context prefill chunking has tradeoffs (8192 regresses vs 4096). We also once caught ourselves benchmarking against a *stale* llama build and re-benched against a current one — the numbers above are the honest, current comparison.
