## TL;DR

`ie` now runs OpenAI's gpt-oss **tensor-parallel** across two consumer Arc Pro B70s —
**both cards compute every token**, not one-at-a-time. This is the engine's first MoE
tensor-parallel path. Three pieces make it work:

1. **MoE expert tensor-parallel** — every expert is split column/row-wise across both
   cards; each card does half of every expert's matmul.
2. **Attention head-shard** — the attention heads (and their KV cache) are split across
   the two cards, so each card holds half the KV and does half the attention.
3. **Host-RAM expert spill** — the 117 B `gpt-oss-120b` doesn't quite fit in 64 GB of
   VRAM at long context, so the experts that overflow a per-card VRAM budget live in
   system RAM and are read over PCIe.

Validated correct on the 20b: two-card perplexity **matches single-card on both axes**
(Phase-2 head-shard: streaming 19.49 vs 19.36, prefill 16.83 vs 16.82) and greedy factual completions are
coherent ("The capital of France is Paris."). On the 120b it now runs end-to-end —
**538–679 tok/s prefill** at typical context, **~31 tok/s decode** (≈2.6× LM Studio's
12.42 on the same two cards), batched-prefill perplexity **bit-identical** to the
token-by-token path (15.19 / 15.20), and coherent chat with working Harmony tool calling.
Full results below.

## Why "both cards compute" is the whole point

The default way to run a model too big for one GPU — LM Studio, and llama.cpp's
`--split-mode layer` underneath it — is **layer-split (pipeline) parallelism**: put the
first half of the layers on card 0 and the second half on card 1. For a single
interactive stream this has a structural problem:

> Card 1 cannot start layer N/2 until card 0 has finished layer N/2−1 **for that same
> token**. There is a hard sequential dependency down the layer stack, and a chat
> session decodes one token at a time. So at any instant **only one card is doing math**
> and the other is waiting its turn. You bought two GPUs and run them like one slow one
> with a hand-off in the middle.

That is exactly what's observed: the LM Studio 120b run pins one card and leaves the
other mostly idle (≈29/32 GB resident, compute mostly waiting).

**Tensor parallelism** splits *each individual matmul* across both cards instead of
splitting *which layers* go where. Every layer, every token, both cards multiply their
half of the weights against the activations, then exchange and sum partial results. There
is no "card 0 finishes, then card 1 starts" — they run the same layer at the same time.
The cost of that is a synchronization (an *all-reduce*) at each split point, which is
where the P2P story below matters.

## The three phases, in accessible terms

The implementation is additive glue over the single-GPU `GptOssModel` (whose math is
lifted verbatim and never edited — the perplexity gate stays green through that untouched
path). It ships in three phases, all committed and validated on the 20b.

### Phase 1 — MoE expert tensor-parallel

gpt-oss is mostly experts (the MXFP4 MoE is the bulk of the weights), so that's where the
split has to happen. For each expert:

- the **gate** and **up** projections are split **column-wise** — the expert's hidden
  width (its feed-forward dimension) is cut in half, so each card produces half of the
  intermediate vector;
- the **down** projection is split **row-wise** along that same halved dimension — which
  means each card produces a *partial* output for the full hidden size, and the two
  partials must be **summed** (the all-reduce).

The router is *replicated*: which experts a token goes to is a deterministic argmax, so
it's computed once on card 0 and the identical routing is used on both cards — no
ambiguity, no extra sync. Two careful details: the down-projection split is aligned to
MXFP4's 32-element block boundaries (you can't slice a 4.25-bit block in half), and the
down-projection **bias is added on one card only** — otherwise it would be counted twice
when the two partials are summed.

Result: **one all-reduce per layer**, on the MoE output.

### Phase 2 — attention head-shard

In Phase 1 the attention runs *replicated* (full weights, full heads) on both cards — it
stays in lock-step for free because both cards start from the identical residual. That's
correct, but it wastes the second card on attention and stores the **whole KV cache on
both cards**, which is the binding constraint at long context.

Phase 2 splits the attention heads across the two cards: q/k/v projections column-wise,
the output projection row-wise (a second all-reduce, same pattern as the MoE down-proj),
and the per-head **sinks**, the q/k/v **biases**, and the **KV cache** all sliced to each
card's head range. This **halves the KV cache per card** and splits the attention compute
— the lever that makes a 99 K-token context fit. (It can be turned off —
`IE_GPTOSS_TP_REPLICATE_ATTN` reverts to the Phase-1 replicated attention.)

### Phase 3 — host-RAM expert spill

Even split across 64 GB, the 117 B 120b is tight, and card 0 is also the **display GPU**
— maxing it freezes the desktop. So loading uses a per-card VRAM **budget**: experts are
placed in device memory until the budget is hit, and any further expert weight planes
spill to **pinned system RAM** (`malloc_host`) and are read over PCIe during the forward
pass. The budget reserves headroom (5 GB on the display card, 3 GB on the second by
default; `IE_GPTOSS_TP_RESERVE_GB` overrides). This is the "~90% on VRAM, ~10% in RAM"
arrangement: the spilled experts are slower to read, but they're a minority of the
traffic, and the fast prefill (below) hides a lot of it.

The load log reports the split per card, e.g. `card 0: NN.NN GB VRAM + N.NN GB host-spill`.

## The host-bounced all-reduce: the honest cost on 2× B70

Tensor parallelism needs that partial-sum exchange (all-reduce) at every split point. On
hardware with working peer-to-peer, the cards would DMA partials to each other directly.
**P2P is hardware-blocked on this pair of B70s**, so the all-reduce **bounces through the
host**:

1. each card copies its partial down to system RAM (D→H), overlapped across cards;
2. the host sums them in fp32;
3. the summed result is copied back up to every card (H→D).

That's a fixed, per-all-reduce latency that does **not** shrink with model size. With
Phase 2 on, that's **two host-bounced all-reduces per layer** (attention + MoE).

This is why tensor-parallel is not a free win on every axis:

- **Prefill / long context — wins.** A prefill pass processes hundreds-to-thousands of
  tokens in one shot, so each all-reduce is amortized over a large tile while both cards
  genuinely compute in parallel. The bigger the context, the more the parallel compute
  dominates the fixed sync cost. This is where the engine's already-fast gpt-oss prefill
  (1.9–4.5× vs llama on a single card) compounds with a second card actually working.
- **Small-model decode — loses.** Decoding one token at a time, the per-layer host-bounce
  latency is paid in full with almost no compute to hide behind. For a model like the 20b
  that *already fits on one card*, splitting it across two cards and paying two host
  round-trips per layer per token makes decode **slower** than just running it on one
  card. Tensor-parallel is the right tool when the model **doesn't fit** on one card (the
  120b), not when it does (the 20b).

So the 20b runs tensor-parallel here purely as a **correctness harness**. The decode-tok/s
case for TP is the 120b, where the alternative isn't "one fast card" — it's "doesn't run,"
or "runs layer-split with a card idle."

## Validation (gpt-oss-20b, 2 cards vs 1 card)

The tensor-parallel path is *not* bit-exact to the single-GPU path — the all-reduce sums
partials in a different order, so floating-point results differ in the last bits. It's
validated by perplexity-match, cosine, and greedy-completion match instead. On the 20b
(which fits on one card, so single-card is the ground truth):

| metric                        | 1 card (reference) | 2 cards (TP, Phase-2 head-shard) |
|-------------------------------|-------------------:|--------------------------------:|
| streaming perplexity          | 19.36              | **19.49**                 |
| prefill perplexity            | 16.82              | **16.83**                 |
| greedy factual completion     | "Paris"            | **"Paris"**               |

The small streaming-PPL delta is the expected reduction-order noise, not a correctness
defect — the prefill PPL (a single batched pass, fewer accumulation boundaries) is within
0.01. The engine's separate reference perplexity gate (a different crown model) stays
**bit-exact** because it runs through the untouched single-GPU code.

## 120B results (2× B70, tensor-parallel)

The 120b loads and runs end-to-end on two B70s through this path — correct, and fast on
both axes.

| metric (gpt-oss-120b, 2× B70 TP)                  | value | note |
|---------------------------------------------------|------:|------|
| prefill — typical ctx (≤16k, Phase-1 replicate)   | **538–679 tok/s** | 1.24–1.54× over the head-shard path |
| prefill — long ctx (~65k, chunk 4096)             | **433 tok/s** | chunk 8192 over-subscribes, regresses to 301 |
| prefill — cumulative this session                 | **34 → 679 tok/s** | ~16–20× the start-of-day number |
| decode                                            | **~31 tok/s** (peak 32.05) | vs LM Studio layer-split **12.42** → **≈2.6×** |
| correctness — batched-prefill perplexity          | **15.19** (replicate-attn) / **15.20** (head-shard) | == token-by-token **15.1985**, bit-identical |
| load — card 0 (display) VRAM / host-spill         | 29.3 GB / 3.4 GB | display-safe, no crash |
| load — card 1 VRAM / host-spill                   | 31.1 GB / 0.3 GB | |

*Measurement notes.* Decode is host/communication-bound on this no-P2P box and carries
±15% run-to-run noise; **31 tok/s is the conservative hold, 32.05 the peak**. The
**12.42** reference is owner-reported for LM Studio's llama.cpp layer-split at ~100 K
context on the same two cards — re-confirm at matched context before quoting the ratio as
a hard number. The throughput figures were measured 2026-06-29 and warrant one clean,
idle-box re-bench before print; the correctness numbers (perplexity and the bit-identical
A/B) are config-independent.

### What made it fast

Prefill climbed in three committed, profiling-led steps:

- **pad-M MoE GEMM (restore XMX): 63 → 127 tok/s** (`cc6b23a`). With 128 experts and
  top-4 routing, each per-expert GEMM sees only a handful of rows; padding the small
  experts up to M=32 keeps them on the XMX matmul path instead of dropping to a scalar
  fallback (this is also the fix for bug 2 below).
- **prefill chunk 256 → 4096: 127 → 433 tok/s** (`08cb3a2`). Bigger chunks amortize the
  fixed per-all-reduce host-bounce over more tokens (8192 over-subscribes, → 301).
- **auto-replicate attention at ctx ≤16384: 433 → 538–679 tok/s** (`34b53c3`).
  Replicating attention removes one of the two per-layer host-bounced all-reduces — see
  bug 3.

Decode is host/communication-bound — not quant- or spill-bound — so the lever was the
all-reduce, not the math:

- **async-scatter all-reduce: 14.33 → 32.05 tok/s (2.24×)** (`d3a4aea`).
- **gate+up MoE gemv fusion** (`49afc6c`): the MoE bucket fell 62% → 53% of the decode
  step; decode-path perplexity 17.90 → 17.08 (a *different* harness from the 15.19
  prefill A/B — don't conflate the two). Opt-out `IE_GPTOSS_NO_MXFP4_X2`.

## The three bugs

Getting from "loads" to "usable" meant cracking three problems — two correctness bugs and
one performance bottleneck.

**1. fp16 residual overflow — the chat "!" garbage.** gpt-oss produces unusually large
activations; at the deep layers (around layer 35) a residual exceeds fp16's max (65504),
becomes `inf`, poisons the KV cache, and cascades to NaN — at which point argmax locks on
token 0 ("!") forever. The fix is one clamp in the residual add (`src/ops/elementwise.cpp`,
`s = fmin(fmax(s, -65504), 65504)`, `38ea8f5`). It's an exact no-op for every in-range
model, so the crown perplexity gate stays **6.4527** bit-exact. After the fix the 120b
holds coherent chat — arithmetic (17 × 23 = 391), a correct haiku, and "red / blue /
yellow" list prompts.

**2. small-M XMX MoE corruption — batched-prefill garbage.** With 128 experts and top-4
routing, a batched prefill spreads its tokens so thin that each expert's GEMM runs at a
tiny M (~1–8 rows). BMG's small-M f16 XMX matmul corrupts **every local row except row 0**.
The stable counting-sort packing always places token 0 at row 0 — so token 0 came out
*bit-exact* while every other token was garbled. That asymmetry was the diagnostic that
cracked it. The fix pads small experts up to M=32 before the XMX matmul
(`3c06c32`→`cc6b23a`). The A/B is decisive: batched-prefill perplexity **123 → 15.1985**,
**bit-identical** (NLL 2.721199) to the token-by-token path's 15.1985, with no regression
on the 20b (19.36). Batched prefill is now the default; `IE_GPTOSS_TP_T1_PREFILL` reverts
to the slow one-token path.

**3. comm-bound prefill — the real bottleneck wasn't the math.** Profiling the prefill
step showed the two host-bounced all-reduces (no P2P on this box) are **67.6%** of it —
attention 33.4% + MoE 34.2% — while the actual MoE compute is only 24.4% and attention
compute just 6.1%. The attention all-reduce is pure head-shard overhead, so at moderate
context the engine auto-replicates attention (Phase 1) and drops that all-reduce entirely:
**433 → 679 tok/s**. It auto-selects for ctx ≤16384 (`34b53c3`), reverting to the
KV-cache-halving head-shard at longer context where the memory saving matters more.

### Chat + tool calling, end to end

The Harmony function-calling loop is wired through `ie serve` with zero server changes,
and it round-trips:

> **User:** "What's the weather in San Francisco?" → the model emits a Harmony function
> call `get_weather{"location":"San Francisco"}` (finish = `tool_calls`); the harness
> returns the result and the model continues — "…foggy, 61 °F, 12 mph" (finish = `stop`).
> A no-tools turn answers "Paris" (finish = `stop`), byte-identical to the tools-disabled
> path.

Only user-defined `functions.*` tools are wired so far; the built-in browser/python tools
are a follow-up.
