Root cause of the 2.3× decode gap: Seal dispatches 612 Vulkan compute shaders per token, each preceded by pipeline barriers. GPU executes ~2.2ms of compute per token but wall time is ~58ms — only 3.9% GPU utilization. The other 96% is barrier/submission overhead: each barrier round-trip is ~88µs, but each kernel runs only ~4µs. llama.cpp builds the entire token's computation graph into a single command buffer and submits it atomically — far fewer synchronization points.
Prefill gap (48×) is worse than decode: llama.cpp uses a tiled flash attention kernel for T>1 (prefill), reducing attention complexity. Seal uses naïve O(T × ctx) attention for prefill — every attention matrix multiply gets a separate dispatch with barriers. At 512 tokens this is ~262,000 serial attention ops. Fix: implement tiled prefill attention (flash-attn T>1 path).

4. KV Cache Implementation

Seal KV Cache

struct KvCache {
    // Per full-attn layer (10 layers):
    k: Vec<f32>  // [max_seq, n_kv_heads, head_dim]
    v: Vec<f32>  // [max_seq, n_kv_heads, head_dim]
    // Default: FP32 host storage
}

TurboQuant mode (--turboquant-kv):
    4-bit K + 4-bit V packed
    Randomized Hadamard rotation
    Groupsize=16, per-group scale (FP16)
    ~4× compression vs FP32

Write path (SPLIT — BUG):
    Decode T=1:
      kv_append_pair kernel writes new KV
      TurboQuant: quantizes inline
    Prefill T>1:
      ⚠️ KV rows written as FP32 only
      TurboQuant NOT applied during prefill
      → KV cache is FP32 for prompt tokens,
        INT4 only for generated tokens

Read path:
    kv_append_pair reads back for attention
    attention_split reads cached pairs
    No flash attention for T>1 prefill
    Naïve O(T × ctx) attention computation

Management:
    Sequential append only
    No defragmentation
    Cache invalidated on new context start
⚠️ Prefill KV quantization bug: TurboQuant only activates on the decode path. Prefill tokens land in FP32 KV cache regardless of --turboquant-kv flag. Unified write path needed (same as llama.cpp's ggml_cpy approach).

llama.cpp KV Cache

struct llama_kv_cache {
    // Per layer:
    k: [n_kv_heads, max_ctx, head_dim]
    v: [n_kv_heads, max_ctx, head_dim]
    dtype: configurable (default FP16)
}

Default dtype: FP16 (GGML_TYPE_F16)
Quantized options:
    -ctk q8_0  → Q8_0 keys   (1 scale/32 elem)
    -ctv q8_0  → Q8_0 values
    -ctk q4_0  → Q4_0 keys   (more compression)

Write path (UNIFIED — correct):
    T=1 and T>1 BOTH use ggml_cpy:
      ggml_cpy(K_new[T,kv_h,hd],
               k_cache[slot..slot+T, kv_h, hd])
    If dtype=FP16: straight copy
    If dtype=Q8_0: FP16→Q8_0 quantize inline
      1 scale per 32 elements during write
    Same kernel handles both prefill and decode
    → no prefill KV quantization bug

Read path:
    --flash-attn: flash_attn_ext tiles K/V
      inline dequantization during reads
      causal mask inside kernel
      handles T=1 AND T>1 shapes
    Without flash-attn: ggml_mul_mat dequant

Management:
    llama_kv_cache_find_slot() — find free slots
    llama_kv_cache_update()   — defragment
    Cell-based: supports multi-turn reuse
✅ Unified write path means KV quantization is consistent between prefill and decode. No split-path bugs.

5. T=1 (Decode) vs T>1 (Prefill) Handling

Seal

Decode (T=1):
  GEMV path: each weight matrix uses
    dedicated GEMV kernel (M=1)
  Attention: FA-2 split-K custom kernel
    CHUNKS_PER_WG=8 super-chunk amort.
    SLM tiling, ~92% BW efficiency
  KV write: kv_append_pair (correct)
  Dispatch count: ~612 per token

Prefill (T>1):
  GEMM path: ??? — weight multiplies
    use same GEMV kernel even for T>1
    (no M>1 GEMM kernel yet)
  Attention: NAÏVE O(T × ctx)
    no tiling, no flash-attn equivalent
    separate dispatch per attention step
    → 48× slower than llama.cpp
  KV write: appends FP32 rows
    TurboQuant NOT applied ← bug

Key kernel: forward pass calls
  backend.matmul_qXk_gemv() regardless
  of batch size T. No T>1 dispatch path.
  This is why prefill is so slow.

llama.cpp

Decode (T=1):
  GEMV path: ggml_mul_mat with M=1
    dispatches mul_mat_vec_qX_K kernel
    inline dequantization
  Attention: flash_attn_ext (if --flash-attn)
    or naive SDPA (without flag)
    Q[1,h,d] × K_cache^T — trivial causal

Prefill (T>1):
  GEMM path: ggml_mul_mat with M>1
    DISPATCHES DIFFERENT KERNEL (tiled GEMM)
    mul_mat_q4_K — tiled, batched
    FP16×FP16 → DPAS/joint_matrix (coopmat)
  Attention: SAME flash_attn_ext kernel
    shape drives inner tiling
    Q[T,h,d] × K_cache^T — causal mask inside
    O(T × ctx / tile²) with tiling
    → 885 tok/s on pp512

KV write: same ggml_cpy path for T=1/T>1
  Unified — no prefill quantization bug.

Shape dispatch happens transparently:
  ggml_sycl_mul_mat / ggml_vk_mul_mat
  selects kernel at RUNTIME based on M.

6. MoE Expert Routing

Seal — Fused GPU Router

Seal routes MoE entirely on GPU:

ws_x_norm [1, 2048]
  ↓
moe_qx_topk_fused kernel:
  Router matmul (Q4_K GEMV): x × W_router
    → logits [1, 60]  (60 experts)
  Top-2 selection inline
  Softmax + renorm on GPU
  expert_ids [1, 2] INT32
  expert_weights [1, 2] FP32

For top-2 active experts + 1 shared:
  matmul_q4_k_moe_indirect_dp4a_gate_up_swiglu_sg32
    Fused: gather + gate matmul + up matmul
           + SwiGLU activation — 1 dispatch
  matmul_q4_k_moe_indirect_lcpp (down proj)
  moe_accumulate_add_rmsnorm
    Fused: weighted accumulate + add residual
           + RMS norm — 1 dispatch

Total MoE dispatches per token:
  moe_qx_topk_fused:              1 dispatch
  gate_up_swiglu (experts):       1 dispatch
  down proj (experts):            1 dispatch
  accumulate+add+norm:            1 dispatch
  = ~4-6 dispatches for full MoE FFN

Seal advantage: FUSED kernels cut the
per-op overhead that llama.cpp has.

llama.cpp — Graph Nodes Per Expert

llama.cpp routes MoE via graph nodes:

ws_x_normed [T, hidden]
  ↓
ggml_mul_mat(x, W_router[E,H])
  → logits [T, E] FP32

Top-k + softmax + renorm:
  CPU-side for most backends
  (tensor is small: [1, 60])
  expert_ids [T,k] INT32
  expert_weights [T,k] FP32

For each active expert e:
  gather → x_e [n_tok, hidden]
  gate_out = ggml_mul_mat(x_e, W_gate_e)
  up_out   = ggml_mul_mat(x_e, W_up_e)
  h_out    = ggml_mul(ggml_silu(gate_out), up_out)
  down_out = ggml_mul_mat(h_out, W_down_e)
  out += weight_e * down_out

Each of the above = separate graph node
  = separate Vulkan dispatch
  No fused gate+up+silu kernel
  No fused accumulate+norm

For top-2 from 60 experts:
  ~10-15 separate dispatches for MoE FFN

llama.cpp limitation: graph approach means
no mandatory fusion — each op is its own
kernel launch. Seal's fused MoE is better.

8. Dispatch Overhead Analysis

Metric Seal (measured) llama.cpp (estimated)
Dispatches per decode token ~612 (148,716 dispatches / 243 tokens) ~20-60 (graph nodes per token)
Avg GPU work per dispatch 530.4ms / 148,716 = 3.6 µs/dispatch ~50-100 µs/dispatch (estimated)
Total GPU work per token (decode) ~2.18 ms GPU / 56.6 ms wall ~15-20 ms GPU / ~25 ms wall (est.)
GPU utilization 3.9% (530ms GPU / 13,758ms wall) ~60-80% (estimated)
Inter-dispatch barrier overhead ~88µs per barrier (56ms / 612 barriers) ~0.4ms per barrier (25ms / 60 dispatches)
Fence waits per token 1 (all 612 dispatches in one command buffer) 1 (entire graph in one submit)
The 3.6µs-per-kernel problem: Seal's kernels are individually fast (3-5µs each at production shapes), but every kernel is preceded by a full pipeline barrier (VK_PIPELINE_STAGE_COMPUTE_SHADER_BIT). The GPU must drain the previous shader invocation before starting the next. On BMG G31 this round-trip costs ~80-90µs wall-clock even though the GPU only executes for ~4µs. The fix is not to remove barriers (correctness requires them between dependent ops) but to reduce the number of independent dispatch calls by fusing more ops into single kernels — which is exactly what MK-3 and MK-4 megakernels target.

9. Top Opportunities to Close the Gap

# Fix Expected Impact Complexity
P0 LM head kernel routing — matmul_q6_k_gemv_v1_sg32 fires at 90µs avg for the 248,320-vocab LM head. Should route to v2_glsl (3.73µs) or lcpp_full. Fix the shape-dispatch condition in compute.rs. Removes 22ms/token of unnecessary GPU time (4.1% of GPU time on wrong path). Direct 5-8% decode speedup. Low — routing logic change, no shader work
P1 MK-3/MK-4 megakernels — Fuse the SSM layer (~7 dispatches) and the MoE FFN (~4-6 dispatches) into single kernels. Target: reduce 612 dispatches/token to <100. If barrier overhead drops proportionally: ~6× decode speedup → ~100 tok/s range. High — new WGSL/GLSL kernels; cross-WG sync
P2 Tiled prefill attention (flash-attn T>1) — Current naïve O(T×ctx) attention for prefill. llama.cpp's flash_attn_ext handles T>1 with the same kernel. Need a WGSL tiled attention kernel for M>1 prefill. At pp512: Seal currently ~5 tok/s estimated; tiled attn → 100-400 tok/s range. High — new attention kernel for T>1 shapes
P3 Fix INT8/TurboQuant KV prefill bug — Unified write path for T=1 and T>1. Prefill tokens should write quantized KV rows through the same kv_append_pair path as decode. P4 M>1 GEMM kernels for prefill — Weight projections during prefill use the same M=1 GEMV kernels regardless of batch size.
