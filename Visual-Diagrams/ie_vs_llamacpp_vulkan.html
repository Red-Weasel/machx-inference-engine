IE Native Engine vs llama.cpp — Head-to-Head

Model: Qwen3.6-35B-A3B Q4_K_M (lmhead-q4k variant)  |  Hardware: Intel Arc Pro B70 32GB (BMG G31)  |  IE Backend: SYCL/Level Zero  |  llama.cpp Backend: Vulkan  |  Date: 2026-05-01
⚠ Caveat — these numbers are EXPLORATORY, not a controlled head-to-head:
  • Model variant differs: IE used Qwen3.6-35B-A3B-Q4_K_M-lmhead-q4k.gguf (output.weight pre-converted Q6_K→Q4_K via llama-quantize) while llama.cpp used the unmodified Qwen3.6-35B-A3B-Q4_K_M.gguf. The lmhead variant alone gave IE ~+12% decode in prior bench.
  • Workload shape differs: IE numbers are from 5 real prompts of 51–219 tokens via ie-bench-suite; llama.cpp numbers are from synthetic llama-bench -p 512 -n 128 -r 3. Prefill throughput depends strongly on prompt length, so pp51–pp219 vs pp512 is not directly comparable.
  • Headline ratios (1.28× decode, 7.7× prefill gap) inherit both confounders and should not be cited as "engine vs engine" wins/losses without re-running both engines on the identical model artifact + identical prompt schedule.
What the numbers DO support: rough order-of-magnitude positioning of the engine (decode broadly competitive, prefill an order of magnitude behind) — useful as a directional signal, not a benchmark claim.
Setup: Same physical GPU (Arc Pro B70, BMG-G31). IE backend SYCL/Level Zero, kernel-profiler enabled, 3-run median per prompt across 5 fixed prompts. llama.cpp backend Vulkan, llama-bench -p 512 -n 128 -r 3. Both runs isolated, no concurrent processes.

The decode story has flipped: IE is now 28% faster than llama.cpp Vulkan on tg128 across the entire 5-prompt suite (50.4 vs 39.5 tok/s). This comes from custom decode kernels with full memory-pattern control: SG-cooperative Q6_K dequant (won −34% on lm_head and ssm_out paths), algebraic-folded Q4_K GEMV (vec<uint8_t,16> loads with two-FMA-chain accumulation), adaptive CHUNKS_PER_WG attention split-K (~92% BW efficiency), and 5 fusion kernels eliminating redundant kernel launches (residual+RMS, ssm α+β dual, MoE gate+up+SwiGLU, MoE accumulate+norm, shared-expert gate+up dual).
Prefill gap (~7.7×) remains the single biggest issue: llama.cpp uses tiled GEMM (mul_mat_q4_K with M>1 dispatch) and tiled flash-attention (flash_attn_ext) for prefill — both leverage Intel XMX (joint_matrix / coopmat) for FP16 dot products. IE currently calls the same GEMV kernel even for T>1, plus naïve O(T × ctx) attention without flash-attn tiling. The fix is the planned ESIMD 2D-block-load path (PLAN.md item 13/14) which is gated behind the BMG-G31 DEVICE_LOST issue.

4. KV Cache Implementation

IE KV Cache

struct KvCache {
    // Per full-attn layer (10 layers):
    k_fp16:  sycl::half*  // [n_kv_heads, max_ctx, head_dim]
    v_fp16:  sycl::half*  // [n_kv_heads, max_ctx, head_dim]
    // Optional: INT8 quantized arrays
    k_int8:    int8_t*       // half BW vs fp16
    v_int8:    int8_t*
    k_scales:  sycl::half*   // 1 scale per row
    v_scales:  sycl::half*
}

Default dtype: FP16 (per-row, in-place append)
INT8 mode (--int8-kv):
    Symmetric INT8 + per-row FP16 scale
    ~2× compression vs FP16
    Matches llama.cpp's Q8_0 KV approach

Write path (UNIFIED):
    fa2_append_int8 / _fp16:
      append (k, v) for new token at start_pos
      INT8: quantizes inline, writes scale
      Same kernel handles T=1 and T>1
      (fa2_append_int8 fires once per call;
       prefill chunks invoke same kernel with
       T=chunk_size in a loop)

Read path:
    fa2_partial_int8 (decode FA-2 split-K):
      adaptive CHUNKS_PER_WG = max(1, n_chunks/12)
      INT8 dequant inline in compute loop
      tile-cooperative SLM K/V load + scales
      online softmax accumulators (m, l, out)
      ~92% BW efficiency at decode shapes

Management:
    Sequential append by start_pos
    No defragmentation
    Cache reset on new context start
✅ Unified write path: Same FA-2 append kernel handles T=1 decode and T>1 prefill chunks. INT8 quantization fires consistently for both when --int8-kv is enabled.

llama.cpp KV Cache

struct llama_kv_cache {
    // Per layer:
    k: [n_kv_heads, max_ctx, head_dim]
    v: [n_kv_heads, max_ctx, head_dim]
    dtype: configurable (default FP16)
}

Default dtype: FP16 (GGML_TYPE_F16)
Quantized options:
    -ctk q8_0  → Q8_0 keys   (1 scale/32 elem)
    -ctv q8_0  → Q8_0 values
    -ctk q4_0  → Q4_0 keys   (more compression)

Write path (UNIFIED — correct):
    T=1 and T>1 BOTH use ggml_cpy:
      ggml_cpy(K_new[T,kv_h,hd],
               k_cache[slot..slot+T, kv_h, hd])
    If dtype=FP16: straight copy
    If dtype=Q8_0: FP16→Q8_0 quantize inline
      1 scale per 32 elements during write
    Same kernel handles both prefill and decode
    → no prefill KV quantization bug

Read path:
    --flash-attn: flash_attn_ext tiles K/V
      inline dequantization during reads
      causal mask inside kernel
      handles T=1 AND T>1 shapes
    Without flash-attn: ggml_mul_mat dequant

Management:
    llama_kv_cache_find_slot() — find free slots
    llama_kv_cache_update()    — defragment
    Cell-based: supports multi-turn reuse
✅ Multi-turn reuse via cell-based management with defragmentation. IE currently lacks defrag; cache is reset on new context.

5. T=1 (Decode) vs T>1 (Prefill) Handling

IE

Decode (T=1):
  GEMV path: gemv_q4_K + gemv_q6_K
    each weight uses dedicated GEMV kernel
    algebraic-folded inner loop (M=1)
    vec<uint8_t,16> aligned loads
    SG-cooperative Q6_K dequant (huge/med)
  Attention: fa2_partial_int8 split-K
    adaptive CHUNKS_PER_WG, ~92% BW
    online softmax accumulators
    INT8 KV inline dequant
  KV write: fa2_append_int8 (correct, unified)
  Dispatch count: ~280-320 per token
  → 50.4 tok/s steady-state (28% over llama.cpp)

Prefill (T>1):
  GEMM path: gemm_q4_K (M_TILE=8) for Q4_K
    weight reads amortized 8× across rows
    XMX (joint_matrix) selected if N % 64 == 0
    AND K % 256 == 0
    Falls back to scalar gemv_q for Q6_K
  Attention: full_attention_fa2_decode_int8
    same FA-2 split-K kernel as decode,
    looped per token (no T>1 batching)
    NAÏVE for prefill — no tiled flash-attn
  KV write: same fa2_append_int8 path
    Unified — no split path bug.

Prefill gap: gemm_q4_K only handles Q4_K.
  Q6_K weights (attn_v, ssm_out, attn_qkv DN)
  fall back to per-row gemv loop = slow.
  PP @ T=512 ≈ 100-130 tok/s vs 885 lcpp.

Planned fix: ESIMD 2D-block-load path (PLAN.md
  item 13/14) gated behind BMG-G31 DEVICE_LOST
  issue.  Switch from joint_matrix to xmx::dpas
  via ESIMD for direct DPAS control.

llama.cpp

Decode (T=1):
  GEMV path: ggml_mul_mat with M=1
    dispatches mul_mat_vec_qX_K kernel
    inline dequantization
  Attention: flash_attn_ext (if --flash-attn)
    or naive SDPA (without flag)
    Q[1,h,d] × K_cache^T — trivial causal

Prefill (T>1):
  GEMM path: ggml_mul_mat with M>1
    DISPATCHES DIFFERENT KERNEL (tiled GEMM)
    mul_mat_q4_K — tiled, batched
    FP16×FP16 → DPAS/joint_matrix (coopmat)
  Attention: SAME flash_attn_ext kernel
    shape drives inner tiling
    Q[T,h,d] × K_cache^T — causal mask inside
    O(T × ctx / tile²) with tiling
    → 885 tok/s on pp512

KV write: same ggml_cpy path for T=1/T>1
  Unified — no prefill quantization bug.

Shape dispatch happens transparently:
  ggml_vk_mul_mat selects kernel at runtime
  based on M.  Q4_K and Q6_K both have GEMM
  paths.

Critical for prefill: tiled flash-attn for
  T>1 means O(T × ctx / tile²) work, vs
  IE's O(T × ctx) per-token serial loop.
  At T=512 this is the main 7.7× gap source.

6. MoE Expert Routing

IE — Fused GPU Router + Fused Experts

IE routes MoE entirely on GPU:

ws_x_normed [1, 2048]
  ↓
moe_qx_topk_fused kernel (Q4_K GEMV):
  Router matmul: x × W_router
    → logits [1, 256]  (256 experts)
  Top-2 selection inline (sigmoid + topk)
  Softmax + renorm on GPU
  expert_ids [1, 2] INT32
  expert_weights [1, 2] FP16

For top-2 active experts + 1 shared:
  moe_decode_gate_up_silu_q4k:
    Fused: gather + gate matmul + up matmul
           + SwiGLU activation — 1 dispatch
  moe_decode_down_q6k (or _q4k):
    Q6_K weight, indirect-gather GEMV
  Shared expert (always active):
    gemv_q4_K_dual_ffn (gate+up FUSED dual)
    swiglu, ffn_down_shexp
  moe_accumulate_add_rmsnorm:
    Fused: weighted accumulate + add residual
           + RMS norm — 1 dispatch

Total MoE dispatches per token (decode):
  moe_qx_topk_fused:           1 dispatch
  routed gate+up+silu:         1 dispatch
  routed down proj:            1 dispatch
  shared expert gate+up dual:  1 dispatch
  swiglu:                      1 dispatch
  shared down + scaled_add:    2 dispatches
  = ~6-8 dispatches for full MoE FFN

IE advantage: FUSED kernels cut the
per-op overhead that llama.cpp pays.

llama.cpp — Graph Nodes Per Expert

llama.cpp routes MoE via graph nodes:

ws_x_normed [T, hidden]
  ↓
ggml_mul_mat(x, W_router[E,H])
  → logits [T, E] FP32

Top-k + softmax + renorm:
  CPU-side for most backends
  (tensor is small: [1, 256])
  expert_ids [T,k] INT32
  expert_weights [T,k] FP32

For each active expert e:
  gather → x_e [n_tok, hidden]
  gate_out = ggml_mul_mat(x_e, W_gate_e)
  up_out   = ggml_mul_mat(x_e, W_up_e)
  h_out    = ggml_mul(ggml_silu(gate_out), up_out)
  down_out = ggml_mul_mat(h_out, W_down_e)
  out += weight_e * down_out

Each of the above = separate graph node
  = separate Vulkan dispatch
  No fused gate+up+silu kernel
  No fused accumulate+norm

For top-2 from 256 experts:
  ~10-15 separate dispatches for MoE FFN

llama.cpp limitation: graph approach means
no mandatory fusion — each op is its own
kernel launch.  IE's fused MoE is better.

The dispatch story is different from Seal: Seal's 612 dispatches/token at 3.6µs each was almost entirely overhead-bound (3.9% GPU utilization). IE's ~280 dispatches/token at ~50µs average means individual kernels are ~14× more substantial — kernels do real work, and dispatch overhead is a smaller fraction of total time. IE's wins come from custom kernels reaching higher per-dispatch BW efficiency, not from reducing dispatch count to llama.cpp levels.
The fusion strategy: IE's per-step launch count was reduced ~30% across this session via 5 successful fusions: residual+RMS, ssm α+β dual, MoE gate+up+SwiGLU, MoE accumulate+norm, shared-expert gate+up dual. Each fusion targeted a same-input same-shape pair where the underlying kernels could share SLM A loads. Cross-layer fusion was attempted (residual_moe + next_attn_norm) but reverted — no net gain because the same-WG-per-row layout doesn't change cost when split vs combined at this shape. Mixed-dtype fusion (attn_k Q4 + attn_v Q6) was attempted and reverted — only 6 of 10 layers qualified (Q4_K_M policy mix), and doubled per-call compute outpaced the launch-overhead saving.

Summary: IE wins on decode throughput (28% over llama.cpp) via custom per-shape kernels, deeper fusion, and SG-cooperative Q6_K dequant patterns specific to BMG-G31. llama.cpp wins on prefill (~7.7×), portability, automatic memory planning, multi-turn KV reuse, and CPU fallback safety. The performance gap on prefill is structural: llama.cpp uses tiled GEMM + tiled flash-attention with XMX/coopmat acceleration; IE's planned ESIMD path (PLAN.md item 13/14) is the primary lever to close it but is gated behind a known BMG-G31 DEVICE_LOST issue. Decode parity is achieved; prefill is the next bounded engineering effort, and the cleanest path is reactivating ESIMD.
